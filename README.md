# MedRAG India Production Rebuild

This workspace now contains a production-oriented MedRAG stack:

- `apps/api`: FastAPI, SQLAlchemy, Alembic, Postgres, LangGraph, Qdrant, Redis/RQ,
  S3-compatible object storage, ClamAV scanning, OCR/ingestion workers, consent,
  care-team access, immutable audit, answer traces, agentic care coordination,
  hospital management, consultation booking, and LoRA/QLoRA fine-tuning support.
- `apps/web`: React + TypeScript frontend served by Vite in development and a
  small Node/Express server in production.

## Why Not Keep The Shell Scripts

The original scripts are useful prototypes, but they are risky as the foundation:

- Application code is generated by shell heredocs/base64 snippets.
- Secrets are written into shell profile files.
- Database schema is patched directly instead of migrated.
- CORS and auth defaults are too permissive for patient data.
- UI, API, OCR, RAG, and agent behavior are tightly coupled.

## Recommended AI Architecture

Use **fine-tuned model + RAG**, with **LangGraph** for clinical workflows and
**LangChain** inside graph nodes.

LangGraph is the better fit for MedRAG because clinical flows need branching,
durable state, auditability, human review, and later resume/retry behavior.
Plain LangChain chains are simpler and good for predictable two-step RAG, but
they are less suitable once the workflow becomes: safety check -> retrieval ->
generation -> validation -> audit trace -> possible clinician approval.

Fine-tuning and RAG solve different problems:

- Fine-tuning teaches the model your preferred behavior: Indian clinical phrasing,
  SOAP note style, safe refusal patterns, prescription-summary format, multilingual
  tone, and how to use citations.
- RAG supplies current facts from clinical guidelines and verified patient documents.

For medical QA, never rely on model weights alone. BioMistral should still
receive retrieved context at inference time, whether it is the Hugging Face base
model or a future fine-tuned adapter.

## Medical Data Safety Policy

The backend now enforces these safety filters:

- Patient AI mode is education-only. Patients are refused diagnosis and prescription
  requests and are redirected to clinician consultation or emergency care when needed.
- Doctor/hospital AI mode is clinical decision support. It may help with diagnosis
  and prescribing, but only for licensed users with care-team membership and active
  patient consent.
- Clinical staff cannot download/export patient documents. A hard backend policy
  blocks download attempts unless the actor is the patient.
- X-rays, dental images, and symptom photos are stored and scanned, but patient
  AI does not diagnose images. BioMedCLIP image embeddings are stored in a
  separate Qdrant image collection for similarity/retrieval support only.
  Optional vision output is preliminary only; a clinician must verify findings
  before image-derived text can enter clinical text RAG.
- Printed reports and printed prescriptions can be stored as raw OCR text for
  human verification. Suspected handwritten prescriptions are marked
  `handwriting_human_transcription_required`; raw machine OCR text is not stored
  in the DB unless `OCR_STORE_HANDWRITTEN_RAW_TEXT=true`.
- Clinical access is minimum-necessary and consent-aware.
- Common direct identifiers such as Aadhaar-like numbers, ABHA-like IDs, phone
  numbers, and emails are redacted from stored answer traces.
- Every clinical AI request writes a hash-chained audit event and an answer trace.
- Responses include no-store/no-cache security headers to reduce browser/proxy leakage.

This is HIPAA-aligned architecture. A real regulated deployment still needs signed
BAAs/vendor agreements, infrastructure encryption/KMS, backup retention controls,
operational policies, staff training, access reviews, incident response, and legal review.

## Use Hugging Face BioMistral With RAG

For the POC, pull BioMistral directly from Hugging Face and use it with RAG:

```env
MODEL_PROVIDER="local_hf"
BASE_MODEL_NAME="BioMistral/BioMistral-7B"
FINETUNED_ADAPTER_PATH=""
LOCAL_MODEL_LOAD_IN_4BIT="true"
```

Runtime flow:

```text
question -> safety graph -> query router -> query rewriting
-> BGE-M3 Qdrant/BM25/RRF retrieval -> BGE reranking -> parent context expansion
-> evidence compression -> prompt BioMistral from Hugging Face
-> citation validation -> answer with trace
```

## Agentic Care Coordination

The care agent is intentionally bounded. It does not diagnose and it cannot call
arbitrary tools. It can only use approved internal actions:

- `POST /care-agent/yearly-health-scan`: creates a patient calendar event and
  health task for an annual preventive scan.
- `POST /care-agent/symptom-action`: assesses symptom severity and either books
  a doctor appointment or creates an emergency dispatch request.

## Hospital Management + Consultations

Hospital administrators can create hospitals, departments, assign doctor users,
and publish consultation slots:

```text
POST /hospitals
POST /hospitals/departments
POST /hospitals/doctors
POST /hospitals/slots
```

Patients can search hospitals/slots and book a consultation:

```text
GET /hospitals?city=Bengaluru&speciality=General%20Medicine
GET /hospitals/slots?speciality=General%20Medicine
POST /hospitals/consultations/book
```

The booking creates a confirmed `appointments` row with hospital, department,
doctor, slot, consultation mode, urgency, reason, and booking reference. Staff can
update statuses such as confirmed, checked_in, completed, cancelled, and no_show.

Critical symptom escalation is based on safety red flags and severity. In the
demo, ambulance dispatch returns an auditable provider reference. In production,
replace `AmbulanceDispatchService` with an approved EMS/108/112/hospital command
center integration and require location verification, patient confirmation where
possible, and emergency operations review.
